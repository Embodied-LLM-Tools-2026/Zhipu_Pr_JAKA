# RSS 2025 投稿级系统改进方案
## VLA-as-Function 的契约化混合控制系统

### 目标
将当前 demo 从“可运行的集成系统”升级为 RSS 可接受的系统论文：
核心思想是将 VLA 降级为具备明确契约（Tool Contracts）的原子技能函数，并通过 **Router + Verifier + Hierarchical Recovery** 构建一个 **可控、可验证、可恢复** 的长程机器人系统。

---

### 0. 核心定位变革（Paradigm Shift）

#### 0.1 当前状态（Before）
*   一个包含 VLM / Planner / Robot 的集成系统
*   支持语音 → 感知 → 导航 → 抓取 → 递交
*   具备 step-wise tool execution 的闭环雏形

#### 0.2 论文级定位（After）
*   一种基于 **Tool Contracts** 的混合控制架构（Contract-based Hybrid Control）

#### 0.3 核心主张（必须写进 Introduction）
*   ❌ **End-to-End VLA**
    *   缺乏长程任务结构化推理
    *   难以验证是否真的改变了物理状态
*   ❌ **Naive LLM Agent**
    *   缺乏物理可执行性
    *   容易“以为成功”（hallucinated success）
*   ✅ **我们的解决方案**
    *   将 VLA 明确降级为 **短时、局部、接触密集** 的 skill function
    *   所有技能（传统 or VLA）统一受 Tool Contract 约束
    *   由符号化 Agent 通过 Router + Verifier + Recovery 进行调度

> **关键立场（非常重要）**
> *   VLA 不是 agent
> *   VLA 不负责任务推进
> *   VLA 是一个 **被调用、被验证、可回退** 的技能函数

### 2. Tool Contracts：从工程接口到理论定义（P0 必须）

#### 2.1 Tool Contract 的形式化定义
对于任意技能 $s$，定义其 Tool Contract 为四元组：

$$C(s) = \langle P_{pre}(s), C(s), V_{post}(s), F(s) \rangle$$

其中：
*   **Preconditions $P_{pre}$**：执行技能前必须满足的物理或信息状态
*   **Constraints / Invariants $C$**：执行过程中必须始终满足的安全与物理边界
*   **Post-Verifiers $V_{post}$**：技能执行后，通过独立观测验证物理状态是否发生预期改变
*   **Failure Taxonomy $F$**：有限枚举的失败类型，用于触发确定性 Recovery

> ⚠️ **当前系统最缺失的部分**：Failure Taxonomy $F$

#### 2.2 ExecutionResult → Tool Contract（工程对齐）
你现有的：
```python
ExecutionResult(
  status,
  node,
  reason,
  elapsed,
  evidence
)
```

升级为论文级接口（最小改动）：
*   新增 `failure_code: Enum`
*   新增 `verified: bool`（或放入 evidence）

**判定规则（论文中必须明确）：**
一个技能执行成功 iff：
1.  Preconditions 成立
2.  Constraints 未违反
3.  Post-Verifier 返回 True
否则视为失败，并返回结构化 `failure_code`。

#### 2.3 论文必须出现的核心表（Method 章节）

| Category | Skill | Preconditions $P_{pre}$ | Verifier $V_{post}$ | Failure Codes | Recovery Strategy |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Perception** | `search_area` | Robot localized | Target visible > $\tau$ | `NO_OBSERVATION` | Move to next viewpoint |
| **Mobility** | `navigate_area` | Map valid | Arrival error < $\epsilon$ | `NAV_BLOCKED` | Backoff + retry |
| **Manipulation** | `predict_grasp` | RGB-D valid | IK reachable | `ZEROGRASP_FAILED` | Adjust view |
| **Manipulation** | `execute_grasp` | Grasp pose valid | Gripper force + vision | `GRASP_FAIL` | Re-grasp |
| **VLA** | `vla_grasp` | Target in workspace | Delta state verified | `VLA_NO_EFFECT` | Fallback to classic |

这张表完成，论文完成一半。

### 3. 显式 Verification Loop（P0 必须）

#### 3.1 为什么必须显式验证
*   不能假设“动作执行完 = 成功”
*   RSS 极度反感 implicit success assumption

#### 3.2 执行流程（论文级描述）
`Action` → `Wait` → `Verify` → `Result`

#### 3.3 最小可接受的 Verifier 集合（先做这 3 个）
1.  `verify_target_visible`
2.  `verify_pose_ready`
3.  `verify_grasp_success`

即使 verifier 是 heuristic，只要 **存在 + 被调用 + 被统计**，就是系统论文，而不是 demo。

### 4. Failure Taxonomy（P0 必须）

将所有字符串 `reason` 映射为有限枚举：

#### 4.1 示例分类
*   **PERCEPTION**
    *   `NO_OBSERVATION`
    *   `RGB_UNAVAILABLE`
    *   `DEPTH_LOCALIZATION_FAILED`
*   **MOTION / NAV**
    *   `NAV_BLOCKED`
    *   `ROTATE_FAILED`
*   **MANIPULATION**
    *   `ZEROGRASP_FAILED`
    *   `IK_FAIL`
    *   `GRASP_SLIP`
*   **INFRA**
    *   `GRIPPER_UNAVAILABLE`
    *   `NAVIGATOR_UNAVAILABLE`

Failure Code 是后续 **Recovery、实验统计、消融分析** 的核心支点。

### 5. Hierarchical Recovery（P0 必须）

#### 5.1 Level 1：Reflexive Recovery（不经过 LLM）
*   `IK_FAIL` → base 微动 → retry
*   `NAV_BLOCKED` → backoff → retry
*   `GRASP_SLIP` → open → lift → reset
*   **特点**：毫秒级、确定性、低成本

#### 5.2 Level 2：Local Re-planning（局部调整）
*   连续 N 次 L1 失败
*   `TARGET_LOST`
*   **动作**：
    *   `search_area` / `rotate_scan`
    *   修改抓取角度、ROI、站位

#### 5.3 Level 3：Global Re-planning
*   超时
*   硬件不可用
*   不可逆状态
*   **动作**：
    *   `ReflectionAdvisor`
    *   重新生成 BT / Plan

> **论文卖点**：Level 1 & 2 显著减少 Level 3（LLM 调用）次数，提高实时性和成功率。

### 6. Skill Router：VLA-as-Function 的关键机制（P1）

#### 6.1 Router 目标
决定 **何时调用传统技能**，**何时调用 VLA 技能**。

#### 6.2 路由原则（规则版即可）
*   长距离移动 / 精确几何对齐 → **Classic Skills**
*   接触密集 / 非刚体 / 难以建模 → **VLA Skill**

**推荐模式**：Classic → 把状态拉进可控局部 → VLA finish
